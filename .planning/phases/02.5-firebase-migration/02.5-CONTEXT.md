# Phase 2.5: Firebase Migration - Context

**Gathered:** 2026-03-02
**Status:** Ready for planning
**Source:** PRD Express Path (tender-juggling-starfish.md)

<domain>
## Phase Boundary

Migrate the persistence layer from Supabase PostgreSQL (via SQLAlchemy/SQLModel) to Firebase Firestore. Supabase is blocked by the Indian government. The clean repository pattern (all DB access through 5 repository files returning plain dicts) makes this migration well-scoped.

**Scope:** Backend-only data layer swap. No frontend changes. No production data to migrate — only re-seed the 45-material catalog.

</domain>

<decisions>
## Implementation Decisions

### Database Engine
- Replace Supabase PostgreSQL with Firebase Firestore (Native mode, `asia-south1` region)
- Use `firebase-admin` SDK (>=6.0.0) for server-side access
- Service account JSON auth via `FIREBASE_CREDENTIALS_PATH` env var

### Model Layer
- Convert SQLModel ORM classes to plain Pydantic BaseModel
- ID fields change from `int` to `str` (Firestore document IDs are strings)
- All foreign key fields (`project_id`, `floorplan_id`) change from `int` to `str`
- Remove relationship fields, `__tablename__`, `table=True`, `sa_type=JSON`

### Repository Layer
- Firestore collections: `projects`, `floorplans`, `materials_pricing`, `generated_boms`, `async_jobs`
- Auto-generated document IDs (not sequential integers)
- Dict return contract preserved
- Non-blocking persistence pattern preserved

### Type Propagation
- All `int` ID params across services, workers, routes become `str`
- Route params: `<int:pdf_id>` → `<pdf_id>`, `<int:job_id>` → `<job_id>`

### Testing
- Firestore emulator (`localhost:8080`) replaces real Supabase in tests
- `reset_firestore_client()` for test isolation
- Delete `tests/test_setup_supabase_schema.py` (irrelevant)

### Claude's Discretion
- Firestore client singleton implementation details
- Timestamp serialization approach
- Seed script data format

</decisions>

<specifics>
## Specific Ideas

- Lazy-initialized Firestore client singleton in `app/core/firebase.py` (mirrors existing `database.py` pattern)
- `list_boms_by_project`: 2-step query (get floorplan IDs, then Firestore `in` query, max 30 values)
- Job lifecycle: single-document updates are atomic in Firestore
- Firestore Timestamps → `.isoformat()` in serialization helpers
- Same 45 materials across 10 categories with Indian market pricing in seed script

</specifics>
